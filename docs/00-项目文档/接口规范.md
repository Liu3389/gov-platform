# 接口规范

> 统一前后端接口协议，所有后端服务必须遵守。接口文档使用 Knife4j 4.4.0 自动生成，网关聚合所有服务文档。

---

## 一、RESTful 风格

| 操作 | 方法 | URL 示例 | 说明 |
|------|------|----------|------|
| 查询列表 | GET | `/api/v1/users` | 支持分页参数 |
| 查询详情 | GET | `/api/v1/users/{id}` | |
| 新增 | POST | `/api/v1/users` | |
| 修改 | PUT | `/api/v1/users/{id}` | |
| 删除 | DELETE | `/api/v1/users/{id}` | 逻辑删除 |
| 批量删除 | DELETE | `/api/v1/users/batch` | 传 ID 数组 |

### URL 规范

- 全部小写，单词间用中划线 `-` 连接。
- 资源名用复数，如 `/users`、`/items`。
- 动作通过 HTTP Method 表达，不在 URL 中写动词。
- 网关统一前缀：`/api/{version}/...`，默认版本 `v1`。

## 二、统一响应体

```json
{
  "code": 200,
  "message": "success",
  "data": {},
  "timestamp": 1718888888888,
  "traceId": "abc123"
}
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| code | int | 业务状态码，200 为成功 |
| message | string | 提示信息 |
| data | object/array | 业务数据 |
| timestamp | long | 时间戳 |
| traceId | string | 链路追踪 ID |

### 分页响应

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "records": [],
    "total": 100,
    "pageNum": 1,
    "pageSize": 10
  }
}
```

### 常见错误码

| 错误码 | 含义 |
|--------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未登录/Token 无效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 500 | 系统内部错误 |
| 1001 | 业务自定义错误 |

## 三、Knife4j 注解规范

### 3.1 Controller 示例

```java
@Tag(name = "用户管理")
@RestController
@RequestMapping("/api/v1/users")
public class UserController {

    @Operation(summary = "用户登录", description = "支持手机号+验证码或账号密码")
    @PostMapping("/login")
    public Result<LoginVO> login(@RequestBody @Valid LoginDTO dto) {
        return Result.ok(userService.login(dto));
    }

    @Operation(summary = "根据ID查询用户", description = "返回用户详情")
    @Parameter(name = "id", description = "用户ID", required = true, example = "10001")
    @GetMapping("/{id}")
    public Result<UserVO> getById(@PathVariable Long id) {
        return Result.ok(userService.getById(id));
    }
}
```

### 3.2 必须注解清单

| 元素 | 注解 | 必填 |
|------|------|------|
| Controller | `@Tag(name = "xxx")` | 是 |
| 方法 | `@Operation(summary, description)` | 是 |
| 路径参数 | `@Parameter` | 是 |
| 请求体参数 | `@Schema(description, example)` | 是 |
| DTO/VO | `@Schema(description)` | 是 |

### 3.3 DTO / VO 示例

```java
@Schema(description = "登录请求参数")
@Data
public class LoginDTO {

    @Schema(description = "用户名", requiredMode = REQUIRED, example = "admin")
    @NotBlank(message = "用户名不能为空")
    private String username;

    @Schema(description = "密码", requiredMode = REQUIRED, example = "123456")
    @NotBlank(message = "密码不能为空")
    private String password;
}
```

## 四、接口安全规范

### 4.1 统一鉴权

- 登录接口返回 JWT Token。
- 其他接口在 Header 中携带：`Authorization: Bearer {token}`。
- 网关统一校验 Token，白名单接口除外（如登录、注册、Knife4j 文档）。

### 4.2 接口签名（对外接口）

对外暴露的接口必须使用签名机制：

1. 客户端生成时间戳 `timestamp`、随机字符串 `nonce`。
2. 按参数名 ASCII 排序，拼接成字符串。
3. 使用 HMAC-SHA256 或 SM3 计算签名 `sign`。
4. 服务端使用相同算法校验签名，并校验时间戳防止重放（5 分钟内有效）。

### 4.3 CA 认证

- 服务端配置 CA 根证书信任库。
- 客户端调用时携带客户端证书。
- 证书私钥禁止上传 Git，使用 Docker Volume 挂载。

### 4.4 敏感数据

- 身份证号、手机号等展示必须脱敏。
- 日志中禁止打印密码、Token、身份证号。

## 五、接口调试入口

### 5.1 Knife4j 地址

- 网关聚合文档：`http://localhost:8091/doc.html`
- 单服务文档：`http://localhost:{服务端口}/doc.html`

### 5.2 调试流程

1. 后端启动服务后，先访问 Knife4j 确认接口已生成。
2. 填写参数，点击“发送”，记录响应。
3. 若异常，截图或复制请求/响应/日志给 AI 分析。

---

## 六、接口变更流程

1. 后端在 `feature/*` 分支修改接口。
2. 同步更新 Knife4j 注解与字段说明。
3. 在 MR/PR 中说明接口变更点。
4. 前端根据 Knife4j 文档调整调用。
